# dd-trace-rust adapter (NATIVE Temper backend — no FFI)

**Status: 151/243 pass, 0 FAIL, 92 skip** (32 ops dd-trace-rs can't express +
60 verified genuine dd-trace-rs v0.5.0 behavior gaps in `KNOWN_RUST_DIFFS`).
Run-then-downgrade honesty model: a listed diff that PASSES still reports PASS,
so the list can't hide a fix.

## Why this one is different

Every other backend either compiles the suite natively (js/py/go/dotnet/java) or
loads it via a C shim (ruby/php). rust is the cleanest of all: **Temper has a
native rust backend**, so the suite is already generated as the
`system-tests-redux` rust *lib crate* (`temper.out/rust/system-tests-redux`).
This adapter is just a rust binary that **path-deps that crate directly**,
implements the suite's `TracerTrait` against the real dd-trace-rs library, and
runs cases in-process. No FFI, no cdylib, no C ABI.

- **Library:** dd-trace-rs `datadog-opentelemetry` v0.5.0 (Cargo git dep;
  not published to crates.io). It is **OpenTelemetry-based**, so the tracer is
  driven through the `opentelemetry` crate API (0.32) + its Datadog exporter.
- **Capture:** spans are exported to a real ddapm-test-agent; `captured_spans()`
  reads them back over HTTP (ureq) from `/test/session/traces`.
- **Runner:** `run.py` runs each case in its own process (env applied before
  dd-trace-rs inits), mirroring the go backend. Canonical propagation-style names
  are translated to dd-trace-rs's short names (as dd-trace-rb needs).
- **Unsupported ops** (manual baggage — commented out in the upstream rust
  parametric client too; OTel attribute/link removal; RC/stats/telemetry) funnel
  through an `UNSUPPORTED` panic that the run loop turns into a SKIP, mirroring
  the ruby/php shim.

## Build & run

```
# one-time: rust toolchain (cargo); the suite crate is generated by build.sh
bash adapters/dd-trace-rust/build.sh        # temper build -b rust + patch + cargo build --release
temper build -b py                          # case registry for run.py
./.venv-ddtrace/bin/python adapters/dd-trace-rust/run.py
```

## What passes (151)

Core spans + meta/metric/resource, datadog/b3/b3multi/tracecontext header
extract+inject (via the OTel propagators dd-trace-rs installs), parenting, flush,
and much of the config readback (via dd-trace-rs's parsed `Config` accessors —
the same the upstream rust parametric client uses).

## Verified genuine dd-trace-rs v0.5.0 gaps (`KNOWN_RUST_DIFFS`, 60)

dd-trace-rs is a young, OpenTelemetry-based tracer; the conformance suite surfaces
DD-specific wire behaviors it has not implemented yet. Each was verified against
the real delivered span (the assertion reads the on-the-wire span, so the failing
value is dd-trace-rs's actual output, not the adapter's), and basic cases in each
family pass — so these are genuine tracer gaps:

- **128-bit trace ids** (`traceids_128bit*`, 15): no `_dd.p.tid` tag emitted
  (verified absent in the delivered span); also prefers tracestate tid and injects
  128-bit form when generation is disabled.
- **OTEL_* env mapping / config surface** (`otel_env_vars`, 12): `Config` doesn't
  reflect `OTEL_SERVICE_NAME`/`OTEL_RESOURCE_ATTRIBUTES`/`OTEL_TRACES_SAMPLER*`/
  `OTEL_LOG_LEVEL`/`OTEL_SDK_DISABLED`/`OTEL_TRACES_EXPORTER`.
- **source-code-integration git tags** (`tracer.sci_*`, 9): `_dd.git.*` not emitted
  from `DD_GIT_*`.
- **tag-based trace sampling** (`trace_sampling.tag*`, 9): being OTel-based,
  dd-trace-rs makes the sampling decision at span *start*, before the matched tag
  is set, so `DD_TRACE_SAMPLING_RULES` tag matchers never match.
- **single-span sampling** (`span_sampling.*`, 7): `DD_SPAN_SAMPLING_RULES` not
  implemented — no `_dd.span_sampling.*` tags.
- **tracestate edge cases** (`headers_tracecontext.*`, 4): duplicate/empty/
  multi-header/OWS vendor-member retention.
- **misc** (4): `_dd.p.dm` mechanism value (-0 vs -4, shared with go/java/ruby/php);
  W3C baggage header not injected (no manual baggage API — as upstream's rust
  client); `DD_TAGS` space-separated form not parsed; ipv6 agent-url readback
  blocked by the runner forcing `DD_TRACE_AGENT_URL`.

These are surfaced findings, not adapter defects — exactly what the conformance
suite is for against an early tracer.
